# 0001. Slug-based access control for Boxes

## Status
Accepted

## Context
A Box (see `CONTEXT.md`) needs to be reachable by whoever the owner shares it with, and by no one else, without requiring accounts, logins, or invite flows — the product goal is frictionless drop-a-link sharing.

## Decision
Access control is delegated entirely to possession of the Box ID. The ID is a 4-word passphrase generated by `secure-password-utilities` (`POST /api/box/create`, `generatePassphrase(4, DEFAULT_WORDLIST)`), used unmodified as the URL slug (`/box/<boxId>`). There is no login, session, ACL, or per-file permission check anywhere in the request path (`src/routes/box/[slug]/[...file]/+server.ts`) — any request that knows the slug can list, upload to, download from, or recursively delete that Box.

## Consequences
- Zero-friction sharing: a link is the entire onboarding flow.
- Security of a Box is exactly the entropy of the passphrase plus how carefully it's kept out of logs, referrer headers, and public channels — anyone who obtains the slug (leaked link, shared screenshot, browser history sync) has full read/write/delete.
- No revocation mechanism: a leaked slug can't be invalidated without deleting the Box outright.
- Retrofitting real auth (accounts, per-user ACLs) later is a breaking change for every existing share link, since the link *is* the credential today.
